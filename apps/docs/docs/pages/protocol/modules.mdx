# Modules

Modules are optional hook contracts that extend slot behavior. A slot's `module` is called on ownership transitions (`buy`, `release`, `liquidate`), enabling custom logic without modifying the core slot contract.

## MetadataModule

The primary module in 0xSlots. It allows the current occupant to attach a URI (metadata) to the slot.

### How It Works

1. A slot is deployed with the `MetadataModule` address as its `module`
2. The occupant calls `updateMetadata(slot, uri)` on the MetadataModule
3. The URI is stored on-chain and indexed by the subgraph
4. Anyone can read the metadata via `tokenURI(slot)` on the MetadataModule

### Use Cases

- **Ad slots** — Occupant sets a URI pointing to ad creative (banner, native, video)
- **Profile slots** — Occupant sets a URI pointing to profile data
- **Content slots** — Any structured data the occupant wants to associate

### SDK Usage

```ts
// Read metadata via subgraph
const { metadataSlots } = await client.modules.metadata.getSlots({ first: 10 });

// Read metadata via RPC (bypasses subgraph)
const uri = await client.modules.metadata.getURI(moduleAddress, slotAddress);

// Update metadata (occupant only)
const hash = await client.modules.metadata.updateMetadata(
  moduleAddress,
  slotAddress,
  "ipfs://QmNewMetadata..."
);

// Get update history
const { metadataUpdatedEvents } = await client.modules.metadata.getUpdateHistory({
  slot: slotAddress,
});
```

## Module Verification

The factory maintains a registry of modules. Modules can be:

- **Verified** — Audited and approved by the protocol. Shows a verified badge in the UI.
- **Unverified** — Any contract can be used as a module, but it won't have the verified status.

The subgraph tracks module verification status:

```graphql
{
  modules(first: 10) {
    id          # module contract address
    verified
    name
    version
  }
}
```

## Mutable Modules

If `mutableModule` is `true` at deployment, the `manager` can propose module changes using the [pending update model](/protocol/tax-system#mutable-tax):

1. Manager calls `proposeModuleUpdate(newModule)`
2. Update applies on next ownership transition
3. Manager can cancel before it applies

## Building Custom Modules

A module contract must implement the slot hook interface. It receives notifications on:

- `buy` — new occupant entered
- `release` — occupant left voluntarily
- `liquidate` — occupant was liquidated

:::warning
Modules execute arbitrary code during slot transitions. Only use verified modules or modules you've audited yourself.
:::
